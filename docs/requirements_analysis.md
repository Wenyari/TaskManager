# 个人任务管理系统 - 需求分析文档 (PRD)

## 1. 项目概述

本项目旨在开发一款高可用、工程化规范严谨的个人任务管理系统。系统采用前后端分离式设计，底层通过 Monorepo 架构实现全栈（前端 Vue3 + 后端 NestJS）的类型共享与高效协同。核心目标不仅是实现任务的生命周期管理，更在于建立一套包含接口文档管理、自动化容灾备份、容器化部署在内的完整软件工程最佳实践。

## 2. 功能需求 (Functional Requirements)

### 2.1 身份认证与安全模块

系统需提供基础但严谨的用户准入机制：

* **账号注册与登录：** 支持用户创建独立账号，并在登录时进行密码校验。
* **凭证颁发机制：** 登录成功后，系统需签发 JWT (JSON Web Token) 作为后续请求的唯一身份凭证。
* **全局拦截校验：** 除登录/注册等公开接口外，所有业务接口必须通过全局拦截器（Guard）进行 Token 校验，拦截未授权访问。

### 2.2 核心任务管理模块 (CRUD)

任务是系统的核心业务实体，需要支持完整的生命周期流转：

| 业务场景 | 需求描述 |
| --- | --- |
| **创建任务** | 用户可新建任务，需包含核心字段：标题、详细描述、优先级、预计截止日期等。 |
| **任务列表与筛选** | 提供任务看板或列表视图；支持按“状态（待办/进行中/已完成）”和“优先级”等维度进行数据筛选。 |
| **状态流转与更新** | 用户可随时修改任务内容，并能够驱动任务状态的流转（如将“待办”标记为“已完成”）。 |
| **任务删除** | 支持对不再需要的任务进行物理或逻辑删除。 |

### 2.3 个性化与偏好设置

* **状态持久化：** 用户的界面状态（如当前的筛选条件、视图偏好）需缓存在本地，保证页面刷新后状态不丢失。
* **主题管理：** 系统需支持全局主题切换（如明/暗模式或多套主题色），通过动态替换全局样式变量实现。
* **多语言支持 (i18n)：** 系统界面需支持中文与英文的双语一键切换，涵盖所有导航、按钮及提示信息。

## 3. 非功能需求 (Non-Functional Requirements)

### 3.1 工程化与架构规范

* **运行环境一致性：** 强制锁定 Node.js 版本为 v20.10.0，杜绝因环境差异导致的构建失败。
* **Monorepo 架构：** 前后端代码整合至同一代码仓库，要求实现底层数据模型和 TypeScript 接口定义的 100% 共享，消除前后端信息差。
* **代码质量控制：** 开启严谨的 TypeScript 类型检查与 ESLint 语法约束，确保代码的健壮性与可维护性。
* **API 契约驱动：** 必须使用 Apifox 作为唯一的接口契约维护平台，遵循“先定义文档，后编写代码”的并行开发流程。

### 3.2 数据容灾与可靠性

* **自动化备份策略：** 必须提供独立的 Node.js 脚本工具，能够以自动或半自动的方式将数据库中的核心数据导出并归档。
* **数据恢复机制：** 需提供对应的恢复脚本，确保在发生意外时，能够通过指定备份文件快速重建数据库环境。

### 3.3 部署与运维指标

* **全面容器化：** 系统不依赖宿主机环境配置，前后端服务与数据库均需通过 Docker 容器化编排。
* **运维隔离：** 保证开发、测试和生产环境的部署流程完全一致，支持通过标准的容器构建、运行和进入指令（build/run/exec）进行故障排查与独立调试。

## 4. 技术栈选型地图 (供评审参考)

* **前端应用：** Vue3 + TypeScript + Pinia (含持久化插件) + vue-i18n + Less
* **后端服务：** NestJS + 认证模块 (Passport/JWT)
* **数据库：** MongoDB (基于 Mongoose ODM)
* **工程化工具：** pnpm workspaces, nvm, ESLint, Apifox
* **运维层：** Docker, Docker Compose, Node.js 原生 API (用于运维脚本)
